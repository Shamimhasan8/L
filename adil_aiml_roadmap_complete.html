RAG systems
Walk me through how you would build a RAG system from scratch for a domain
Answer with your Toolly RAG chatbot: "I built a RAG system for Toolly's AI tools directory. Architecture: (1) Chunk documents with sentence-boundary awareness — not fixed-size. (2) Embed with sentence-transformers (domain-appropriate). (3) Store in Chroma with HNSW index. (4) At query time: embed query, retrieve top-5 with cosine similarity, re-rank with cross-encoder. (5) Augment LLM prompt with context. (6) Evaluate with RAGAS metrics. Key constraint: Toolly needed under 500ms p95 — I used semantic caching for repeated queries, cutting cost 40%."
What chunking strategy did you choose and why? What happens when chunks are too large?
"I chose sentence-boundary chunking at 256 tokens with 20-token overlap. Recursive character splitter as fallback. Too large chunks: context saturation — the LLM gets buried in irrelevant text and retrieval precision drops. I measured this: chunk size 128 → precision@5 = 0.82. Chunk size 512 → precision@5 = 0.61. The overlap handles cross-boundary information."
Your RAG hallucination rate is 15%. What do you do?
Step-by-step: (1) Is it retrieval or generation? Check if retrieved docs contain the answer. If yes → generation problem; if no → retrieval problem. (2) For retrieval: switch to hybrid search (BM25 + semantic), add re-ranking. (3) For generation: add explicit grounding instructions in system prompt, constrain to say "I don't know" when context insufficient. (4) Measure with RAGAS faithfulness metric. (5) For production: add human-in-the-loop for high-stakes queries.
What is the difference between dense retrieval (FAISS) and sparse retrieval (BM25)? When do you use hybrid?
"Dense (FAISS): semantic similarity via embeddings. Catches synonyms, paraphrases. Fails on exact terminology, rare domain jargon. Sparse (BM25): keyword matching, TF-IDF style. Perfect for exact terms, product codes, technical jargon. Fails on semantic synonyms. Hybrid = both. Use hybrid when domain has specialized terminology that embeddings may not capture well — legal, medical, technical product catalogs. Toolly uses hybrid because AI tool names ('GPT-4', 'Mistral') need exact match, not semantic drift."
LLMs and prompting
How does attention work? Explain it without jargon.
"Imagine reading a sentence: 'The trophy didn't fit in the suitcase because it was too big.' When you process 'it', your brain looks back at all previous words and decides 'trophy' is most relevant. That's attention. For every word in the sequence, the model computes a score against every other word: 'how much should I focus on word X when understanding word Y?' These scores become weights — high weight = pay attention. Multi-head means we run this process several times with different learned 'what to look for' lenses simultaneously."
What is the difference between fine-tuning and RAG? When do you choose each?
"Fine-tuning changes the model's weights — bakes in knowledge permanently. RAG retrieves knowledge at inference time — fresh information always available. Choose RAG when: data changes frequently, you need source citations, data is private/sensitive, you want to avoid hallucination on facts. Choose fine-tuning when: you need consistent format/style, domain jargon that keeps failing prompting, high volume justifying upfront cost. My rule: prompt first → RAG if grounding needed → fine-tune only when both have ceiling."
What is LoRA and why does it make fine-tuning affordable?
"LoRA (Low-Rank Adaptation) freezes original model weights and adds small trainable matrices (adapters) alongside. Instead of updating all 7 billion parameters, you train ~0.1% of parameters — only the adapter layers. Why it works: pre-trained weights already capture most knowledge. You're teaching the model a narrow skill on top, not relearning everything. Result: 100x fewer trainable parameters, fits on single consumer GPU, 10x faster training, still achieves 95%+ of full fine-tune quality."
How do you reduce hallucinations without fine-tuning?
"Three approaches: (1) RAG — ground every response in retrieved documents. If the answer isn't in the context, model says 'I don't know.' (2) System prompt constraints — explicitly instruct: 'Only answer from the provided context. If the context doesn't contain the answer, say so.' (3) Output validation — parse model output and verify claims against source. For Toolly, I combine all three: RAG + grounding instruction + confidence threshold that triggers fallback to 'more information needed.'"
System design
Design a Q&A chatbot system that serves 10,000 concurrent users. Walk through your architecture.
"First I'd ask: latency requirement? Streaming or batch? Cost budget? Assuming real-time, 2s p95 SLA, $X/month budget. Architecture: Load balancer → FastAPI pods (horizontal scaling) → Semantic cache (Redis, ~40% hit rate) → Hybrid RAG (BM25 + FAISS) → LLM routing layer (simple queries → GPT-3.5, complex → GPT-4) → Response. Each pod is stateless for easy scaling. At 10K concurrent: 10 pods × 1K concurrent each. Bottleneck is LLM API rate limits — multi-provider fallback solves. Monitoring: Prometheus (latency, error rate), custom metrics (cache hit rate, cost/request)."
How would you reduce LLM API costs by 70% without sacrificing quality?
"Four levers in priority order: (1) Semantic caching — similar queries get cached responses. 40% of queries are repeats → 40% cost reduction. (2) Hybrid routing — classify query complexity. Simple (60%) → GPT-3.5 at 10x lower cost. Complex (40%) → GPT-4. Combined: 70-75% cost reduction. (3) Prompt optimization — reduce average token count. Shorter system prompts, compressed context. (4) Response streaming — don't wait for full response, stream and stop early when answer complete. Result: from $0.05/query to $0.015/query while quality measured within 2% on eval set."
Your LLM response time suddenly spikes from 800ms to 4s. How do you debug?
"Step 1: Which component? Check timing breakdown: embedding generation, retrieval, LLM call, post-processing. Most likely: LLM API latency spike or retrieval degradation. Step 2: Is it all requests or specific? If 90th percentile spikes but median fine → tail latency issue. If all → systemic. Step 3: Check LLM provider status page. Step 4: Check input lengths — longer prompts → slower. Any prompt change deployed recently? Step 5: Check vector DB query latency — index might need rebuilding. Step 6: Add timing spans to every component if not already there. For Toolly, I pre-built this with Prometheus: separate histograms for each pipeline stage."
ML fundamentals + behavioral
What is an embedding? Explain it to a non-technical co-founder.
"Imagine every word, sentence, or document is a point in a huge coordinate system — like a map with thousands of dimensions. 'Similar things' are placed near each other on this map. 'Dog' and 'puppy' are neighbors. 'Dog' and 'finance' are far apart. Embeddings are the coordinates of that map. When you search Toolly's AI tool directory by typing 'help me write emails,' we convert your query into coordinates, then find all tools whose coordinates are nearby — even if they don't use the exact same words. That's semantic search."
Tell me about a project you built that failed. What did you learn?
Use the Bengali ASR/diarization story: "I built a speaker diarization system targeting DER under 0.20. My first approach was off-the-shelf pyannote — got 35% DER on Bengali audio. Failed because the model was trained on English/European speech. I pivoted to domain adaptation: kept pre-trained speaker embeddings, fine-tuned only the clustering head on Bengali data. Got to DER 0.19. But inference was 8x real-time — too slow for production. Applied knowledge distillation: 56% inference speedup, DER maintained. Lesson: never assume off-the-shelf tools work on your domain. Always benchmark on YOUR data first."
