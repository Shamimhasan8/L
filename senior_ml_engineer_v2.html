Q3 — the MLOps filter (80% of candidates fail here)
"Model in production. 3 months later, accuracy drops 85% → 62%. Walk me through your debugging."
Wrong answer: "I'd retrain the model." Right answer: structured root-cause analysis BEFORE any retraining.
The exact 5 steps that pass this filter:

Step 1: Check monitoring dashboards — what TYPE of degradation? Data drift (input distribution changed)? Concept drift (relationship between features and target changed)? Label drift (class distribution shifted)? The type determines the solution.

Step 2: Inspect feature distributions — compare production statistics vs training statistics. Look for schema changes, new null patterns, range violations. Out-of-distribution inputs?

Step 3: Segment the errors — is accuracy dropping uniformly or only for specific subgroups? If only new users or only certain audio lengths are degrading, that's a different root cause than uniform degradation.

Step 4: Check upstream data pipelines — has anything changed in data collection? A logging bug, schema migration, new data source? 40% of production "model failures" are actually pipeline failures.

Step 5: Only then consider retraining. And even then: train a challenger model, run A/B test on 10% traffic, validate before full rollout. Never blind retrain and replace.

Architecture based on answers:
Hot path: Audio ingestion → VAD (lightweight) → 30s chunk buffer → ASR inference → streaming output.
Diarization path (parallel): Speaker embedding extraction → clustering → attribution merge with transcript.
Scale: Horizontal pod autoscaling on GPU nodes. Kafka queue to decouple ingestion from inference. Model loaded once per pod, not per request.

Trade-offs I'd name explicitly:
— Whisper-medium (1.5GB) vs small (244MB): accuracy vs memory trade-off
— GPU at $0.90/hr handles ~40 concurrent streams vs CPU at $0.10/hr handles ~4
— Streaming adds complexity but reduces perceived latency by 60%

Monitoring I'd set up: Per-stream confidence score distribution, latency p50/p95/p99, queue depth, GPU utilization, error rate by audio quality band.

Do we have labeled Bengali harmful content data, and how much? Multi-label (violence + hate speech + spam) or binary? Bengali script only or Romanized Bengali too?

Approach A — Rule-based + lightweight classifier: Keyword matching → mBERT binary classifier for ambiguous cases. Latency: ~30ms. Cost: $0.0001/post. Weakness: misses nuanced content, high false negatives on new patterns.

Approach B — Fine-tuned multilingual LLM: XLM-R fine-tuned on Bengali hate speech. Latency: ~150ms. Cost: $0.001/post. Weakness: 10x cost, needs labeled Bengali training data we may not have.

My choice: Two-stage pipeline. Stage 1: fast rule-based filter (30ms) routes 60% of posts. Stage 2: fine-tuned classifier only for ambiguous posts. Net latency: ~65ms average. Net cost: 40% of full LLM approach. Human review queue for high-confidence harmful content before action is taken.

With more time: Active learning loop — human reviewers label edge cases that go back into training. Per-script calibration (Bengali script vs Roman Bengali have different patterns). Monitoring: false positive rate by content category and user demographic.
