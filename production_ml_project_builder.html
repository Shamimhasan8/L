Phase 1
Problem definition & business context
Everything before you open a notebook. This is what separates production engineers from tutorial followers.
0/10
Project brief write before coding
Write the business problem (not "predict X" — "reduce/increase X by doing Y")
e.g. "Reduce food waste 25% by predicting demand 24h ahead for restaurants"
Document the current baseline — how is the problem solved today?
Quantify it: manual process = X hours/week, existing tool = Y% error rate
Define why this needs ML — can a SQL query or 3-line heuristic solve it?
If yes, don't use ML. Justify the complexity before adding it.
Success metrics hierarchy
Define the primary business metric (revenue, cost, time saved — not accuracy)
Define secondary ML metrics (precision, recall, RMSE — chosen from business context)
Define guardrail metrics you cannot sacrifice (fairness, latency, interpretability)
Stakeholders & risk
Map stakeholders: end users, decision makers, maintainers, data providers
Each group needs something different. Address all of them.
Document deployment scenario: batch / real-time API / streaming / edge
This determines your entire architecture. Decide now.
Write the failure mode — what happens when predictions are wrong?
Healthcare: false negative = catastrophic. Spam filter: false positive = annoying.
Risk assessment: data risks, model risks, system risks, business risks — with mitigation
Recruiter question you must answer
"What could go wrong with this system?" — You need a prepared, detailed answer before your first commit.
Phase 2
Data strategy & pipeline design
Raw data is sacred. Never modify originals. Every stage versioned, reproducible, documented.
0/12
Data acquisition ethics check required
Identify all data sources — primary + supplementary (weather, holidays, etc.)
For each: access method, update frequency, cost, reliability, legal constraints
Ethics check: robots.txt, terms of service, GDPR/CCPA, rate limits, attribution
Recruiters will ask about data ethics. Many candidates fail here.
Data is messy — not from a clean Kaggle dataset. Collect your own or use real APIs.
Data quality framework — 5 dimensions
Completeness: what % missing is acceptable? Which fields are critical vs optional?
Validity: valid ranges, acceptable formats, consistency rules
Consistency, accuracy, and timeliness checks built into the pipeline
Data versioning with DVC — version raw snapshots, features, train/test splits
Goal: reproduce any experiment exactly 6 months from now
Feature engineering (hypothesis-driven)
Every feature has a written hypothesis: "I believe X will help because Y"
Not random features hoping something works. Each one justified.
Temporal features (trends, seasonality, recency), aggregations, interactions, encodings
Every feature is explainable AND monitorable for drift
If you can't explain it — remove it. If you can't monitor it — it's a risk.
Pipeline architecture
Modular pipeline: Raw → Validated → Processed → Features → Model-ready (each testable separately)
Pipeline is automated, scheduled, monitored, and documented — zero manual steps
Phase 3
Experimentation framework
Random hyperparameter tuning is not experimentation. Form a hypothesis, design the experiment, analyze, repeat.
0/10
Baseline first — always
Statistical baseline: mean/median prediction, last value, moving average, simple heuristics
Simple ML baseline: linear regression, logistic regression, decision tree (default params)
Document baseline performance on ALL metrics — not just accuracy
If complex model beats linear by 2%, question whether complexity is worth it.
Hypothesis-driven experiments
Every experiment has: Hypothesis → Prediction → Design → Results → Conclusion
e.g. "Adding weather features will reduce RMSE by 10% because customer behavior changes with weather"
Feature experiments (add/remove groups), model experiments (families, complexity), data experiments
Track every experiment: git hash, data version, ALL config, results, compute cost, artifacts
Model selection criteria
Choose the SIMPLEST model that meets business requirements (not highest accuracy)
Simpler = easier to debug, explain, retrain, and more robust to distribution shifts
Document models tried, performance comparison, and trade-off reasoning in README
Error analysis
Identify failure patterns: where does model perform worst? Random or systematic?
Investigate root causes and generate improvement hypotheses from error patterns
This shows you don't just tune hyperparameters — you think about problems
Phase 4
Model development & validation
Test set accuracy is not enough. Multiple validation approaches to actually trust your model.
0/10
Train/validation/test strategy time-series critical
Time-series data: NEVER random split. Time-based: oldest 60% train / next 20% val / recent 20% test
Random split = data leakage = model fails in production. This is an instant rejection signal.
Non-temporal: stratified K-fold CV, group splits if data has clusters (same user/location)
Validation strategy must simulate production reality — validate across the same dimensions you'll serve
Metrics suite (never just one metric)
Choose metrics based on business context — not default metrics from the library docs
Imbalanced: AUC-ROC, Precision/Recall/F1. Regression: RMSE vs MAE vs MAPE decision documented.
Check CV variance — wildly different fold scores = unstable model or data heterogeneity
Statistical significance — is improvement over baseline real or random noise?
Robustness & interpretation
Robustness: test with missing features, distribution shift, adversarial inputs, data corruption
Global interpretation: feature importance, partial dependence plots — what drives predictions?
Local interpretation: SHAP values or LIME — why this specific prediction?
Document interpretation findings: "Model primarily relies on X features (Y% importance)..."
Shows you don't train models blindly.
Phase 5
Deployment architecture
Model working in Jupyter notebook means nothing. Production = reliable, scalable, maintainable, observable.
0/10
Deployment pattern decision
Chose deployment pattern based on latency — batch / real-time API / streaming / edge
Default for most student projects: batch or simple API. Don't overcomplicate.
Architecture documented: User → API → Input validation → Feature engineering → Model → Response
API design (if serving real-time)
Input schema: clear required/optional fields, data type validation, range validation
Output schema: prediction value, confidence/probability, model version, timestamp
Error handling: 400 bad input, 500 server error, 429 rate limit — with helpful messages
Authentication, rate limiting, input sanitization in place
Containerization & deployment checklist critical for hiring
Dockerized — works on any machine, dependencies at exact versions, health check included
Do not include training data, dev tools, or secrets in container.
Live demo deployed: Streamlit Cloud / Hugging Face Spaces / Railway / Heroku
Working demo is 10x more valuable than perfect code nobody can run.
Rollback plan defined — what if this deployment breaks things?
Cost estimated — know what it costs to run (compute, storage, API calls)
Phase 6
Monitoring & maintenance strategy
If you don't monitor it, it will break silently. This is where 80% of candidates fail at the interview.
0/8
What to monitor — 4 categories
Business metrics: revenue impact, cost savings, user satisfaction, conversion rates
Model performance: accuracy, precision/recall, RMSE — tracked over time
Data quality: missing value rates, out-of-range values, feature distributions, schema violations
System health: request latency, error rates, throughput, resource utilization
Drift detection & retraining
Detect all 3 drift types: covariate shift (feature distributions), concept drift (target relationship), label shift
Alert if feature mean shifts >2 std devs or error rate increases >10%
Retraining strategy defined: scheduled / triggered / hybrid — with staged rollout (small % first)
Alert conditions defined: Critical (>5% error rate, >500ms latency), Warning (accuracy drop >5%), Info
Incident response plan written: data pipeline failure / model degradation / system outage / security
The interview answer that separates you
When asked "your model's accuracy drops from 85% to 62% in production — walk me through your debugging process", you now answer: check monitoring dashboards for drift type → inspect feature distributions → segment errors → check upstream pipelines → only then retrain → A/B test before full rollout.
Final check
Fatal mistakes & interview prep
15 mistakes that get you rejected + your project IS your interview prep. Every decision needs a ready answer.
The 15 fatal mistakes — confirm you've avoided all
1. No business context
Fix: Always frame as "solves [problem] by [approach], delivering [quantified value]"
2. Only accuracy reported
Fix: Report multiple metrics, confusion matrix, precision/recall trade-off, business impact
3. No baseline comparison
Fix: Start with simple baseline, show ML's improvement justifies the complexity
4. Random split for time-series data
Fix: ALWAYS time-based splits. Never use sklearn's train_test_split on temporal data.
5. Data leakage
Fix: Split data FIRST. Fit preprocessing on train only. Ask: "Can I know this feature before prediction?"
6. Hardcoded paths & parameters
Fix: Config files, environment variables, command-line arguments — no hardcoded values
7. No error handling
Fix: Validate all inputs, catch exceptions gracefully, informative error messages, log everything
8. No documentation
Fix: 7-section README, docstrings, code comments, architecture diagrams
9. Dataset too clean
Fix: Collect your own messy data from real APIs — handle quality issues and document them
10. No deployment
Fix: Deploy as API or batch pipeline, Dockerize, add monitoring dashboard
11. Overfitting to test set
Fix: Use separate validation set for tuning. Test set touched ONCE at the very end.
12. No version control history
Fix: Commit regularly with meaningful messages. Commit history = proof of sustained work.
13. Ignoring class imbalance
Fix: 99% accuracy on imbalanced = predicting majority. Use AUC, F1. Try resampling/weighting.
14. No tests
Fix: Unit tests for critical functions, integration tests for pipelines, API endpoint tests
15. Copying without understanding
Fix: Understand every line. Document why you made each choice. Ready to explain in interviews.
Interview Q&A (click to expand your answer)
▼
"Walk me through your most complex ML project"
Structure: Business problem (30 sec) → Data challenges (30 sec) → Approach + model selection with trade-offs (30 sec) → Results and business impact (30 sec) → Deployment + monitoring (30 sec). Total: 2.5 min. Never lead with tools or accuracy. Lead with the PROBLEM.
▼
"Why didn't you use [obvious approach X]?"
Answer format: "I considered X, but given [constraint 1] and [constraint 2], it would have caused [specific problem]. Instead I chose Y because [production reasoning]." — This shows constraint thinking, not tutorial following.
▼
"Your model drops from 85% to 62% accuracy in production. Debug it."
Step 1: Check monitoring dashboards — data drift, concept drift, or label drift? Step 2: Inspect feature distributions — out-of-distribution inputs? Step 3: Segment errors — uniform or specific subgroups? Step 4: Check upstream data pipelines. Step 5: ONLY THEN consider retraining — with A/B test before full rollout.
▼
"How did you handle missing data?"
Answer format: "Weather data had 5% missing values. Tried: dropping rows (loses too much), mean imputation (ignores temporal patterns), forward fill (propagates errors). Chose: interpolation for short gaps, API backfill for longer. Validated: model performance consistent on non-missing holdout."
▼
"Build me a customer support agent (refunds, questions, escalations)"
Don't say "I'd use GPT-4 with a system prompt." Say: First — latency req, cost per query, failure tolerance? Then: intent classification layer before LLM. Refunds: deterministic logic (cheaper, reliable). Questions: RAG + vector DB. Escalations: confidence thresholding + human-in-loop. Monitoring: intent accuracy, response quality, escalation rate.
▼
"What would you improve with more time?"
Answer from your documented future improvements: "Three priorities: 1) Add [specific external data] — likely X% improvement. 2) Implement [specific technique] for [specific reason]. 3) Build [specific infra enhancement]. Would A/B test each to quantify value before committing."
▼
"What's something in ML you've been obsessing over lately?"
Never say "I've been doing Andrew Ng's course." Say something specific you've actually experimented with: a quantization technique, a paper you reimplemented, an agent system you built, a benchmark you reproduced. Show you create problems for yourself beyond courses.
